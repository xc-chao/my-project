# 月度总结技能说明

## 适用场景

当用户提到以下需求时使用本说明：

- 月度总结
- 最近一天总结 / 最近三天总结 / 最近 30 天总结
- 提交记录总结
- 工作汇报 / 周报 / 月报
- 基于 Git 生成总结
- 生成精简版 markdown 工作总结
- 上传总结到语雀

## 目标

基于用户指定的时间范围和本机 Git 提交记录，自动生成一份**精简、按模块归类、可直接复用**的 markdown（markdown风格：社交媒体风‌：
“用活泼的自媒体风格，多用 emoji、短段落、加粗重点，每段不超过 3 行。”） 总结，并在用户确认后支持上传到指定语雀目录。

## 输入约定

用户输入通常包含两部分：

1. **时间范围**：如“最近一天”“最近三天”“最近 30 天”“本月”
2. **补充描述**：用户自己输入的总结文字、业务背景、重点关注模块

处理规则：

- 用户文本用于补充业务语境，不替代 Git 事实依据。
- 如果用户只说“月度总结”，默认按最近 `30 days ago` 处理。
- 如果用户只给出总结文字但未给时间范围，先补问；若上下文明显是“月报”，可直接按最近 30 天处理。
- 如果用户文本与 Git 记录明显冲突，以 Git 记录为主，并在必要时向用户确认。

## 时间范围映射

优先从用户原话中提取时间范围，并转换为 `git log --since` 可用格式：

| 用户表达                         | Git since 参数 |
| -------------------------------- | -------------- |
| 最近一天 / 1天                   | `1 days ago`   |
| 最近三天 / 3天                   | `3 days ago`   |
| 最近七天 / 7天                   | `7 days ago`   |
| 最近 30 天 / 月度总结 / 本月总结 | `30 days ago`  |

如果用户给出更具体的天数，直接按 `N days ago` 处理。

## 执行流程

### 第一步：确定作者

必须先通过本机 Git 配置获取作者名：

```bash
git config user.name
```

处理要求：

- 优先使用 `git config user.name` 返回值作为 `--author`。
- 如果结果为空，再询问用户作者名，不要自行猜测。
- 不要先用 `git config user.email` 代替，除非用户额外要求。

### 第二步：拉取指定时间范围内的提交

基础命令：

```bash
git log --author="用户名" --since="30 days ago" --oneline
```

要求：

- 先获取 oneline 结果，用于快速浏览提交主题。
- 如果没有任何提交，直接告知用户该时间范围内无可总结内容。
- 不要仅凭 oneline 直接出总结，还要继续看文件和 diff。

建议补充获取：

```bash
git log --author="用户名" --since="30 days ago" --pretty=format:'%H%x09%s'
git log --author="用户名" --since="30 days ago" --name-only --pretty=format:
git log --author="用户名" --since="30 days ago" --stat
```

### 第三步：按文件类型统计

先做文件类型分布统计，用于辅助判断本周期工作重心。

推荐命令：

```bash
git log --author="用户名" --since="30 days ago" --name-only --pretty=format: | \
rg -v '^$' | \
awk -F'.' '{if (NF>1) print $NF; else print "[no_ext]"}' | \
sort | uniq -c | sort -rn
```

解释：

- 统计 `vue`、`ts`、`js`、`md`、`css`、`json` 等文件后缀分布。
- `[no_ext]` 代表没有扩展名的文件。
- 该统计只做辅助，不直接决定总结分类。

### 第四步：筛选重点提交

不要把所有小改动都写进总结。需要先筛选重点提交，再按模块聚合。

忽略或弱化以下提交：

- 合并分支
- 纯格式化
- 仅改注释/文案
- 依赖版本微调
- 锁文件微小变更
- 零散样式修补
- 与主功能无关的小修小补

优先关注以下提交：

- 新页面、新模块、新能力
- 影响业务流程的功能改动
- 关键 Bug 修复
- 代码重构、性能优化、工程优化
- 文档体系建设或重要配置升级

### 第五步：分类判断规则

分类分两层：

1. **初判**：根据 commit message 关键词判断
2. **精判**：根据提交前后代码 diff 判断

初判关键词示例：

- `feat`、`add`、`新增`、`支持` -> 倾向“功能开发”
- `fix`、`bugfix`、`修复` -> 倾向“Bug 修复”
- `refactor`、`optimize`、`perf`、`重构`、`优化` -> 倾向“技术优化”
- `docs`、`doc`、`文档` -> 倾向“文档完善”

精判要求：

- 对候选重点提交读取 `git show --stat --name-only <sha>`，确认影响范围。
- 对代表性提交继续看 diff，确认是“新增能力”“流程调整”“问题修复”还是“内部重构”。
- 如果同一模块有多个提交，合并为一个总结点，不要逐条罗列 commit。
- 如果 commit message 含糊，必须通过 diff 再判断，不能想当然归类。

### 第六步：按模块聚合

最终总结只保留 **5-10 个主要工作点**，并按模块归类。

建议聚合方式：

- 功能开发：`5-8` 个点
- 技术优化：`1-2` 个点
- Bug 修复：只保留重要问题

聚合原则：

- 按“业务模块 / 页面 / 功能能力”聚合，不按技术文件逐条展开。
- 同一个页面下多个相关提交，合并成一个模块项。
- 新页面或新模块必须描述其作用，并预留图片位置。
- 小改动不单独成项，只作为某个主模块的补充说明。

## 输出要求

### 内容风格

- 输出必须是 **markdown**。
- 语言精简，避免流水账。
- 总结聚焦结果和价值，不罗列所有提交。
- 每个模块只写用户真正关心的信息：做了什么、解决了什么、涉及哪里。
- 不要写太多技术细枝末节。
- 社交媒体风‌：“用活泼的自媒体风格，多用 emoji、短段落、加粗重点，每段不超过 3 行。

### 输出结构

请严格按以下结构组织：

```markdown
# XXXX年XX月工作总结

## 一、总体概况

- 本周期主要围绕 xxx、xxx、xxx 展开，涉及页面/模块包括 xxx、xxx。
- 共梳理出 X 个主要工作模块，其中功能开发 X 项、Bug 修复 X 项、技术优化 X 项。

## 二、功能开发

### 完成的主要功能

1. [功能/模块 A] - 提交信息简要描述

   - 具体内容：...
   - 涉及页面/功能：...
   - 图片预留：[预留图片：功能 A]

2. [功能/模块 B] - 提交信息简要描述
   - 具体内容：...
   - 涉及页面/功能：...
   - 图片预留：[预留图片：功能 B]

## 三、Bug 修复

### 修复的重要问题

1. [Bug A] - 问题描述和解决方案
2. [Bug B] - 问题描述和解决方案

## 四、技术优化

1. 代码重构：...
2. 性能优化：...
3. 文档完善：...
```

### 写作约束

- “总体概况”里必须带出涉及页面、涉及功能。
- “功能开发”优先写 5-8 个最有价值的模块。
- “Bug 修复”只保留重要问题，避免写碎小修补。
- “技术优化”最多写 1-2 个重点方向。
- 涉及新页面或完整新模块时，必须增加图片占位符：
  - `[预留图片：页面名称-首页]`
  - `[预留图片：模块名称-核心流程]`

## 总结生成策略

### 总体概况生成方法

从以下信息归纳：

- 核心模块名称
- 代表性提交的页面与功能点

输出目标：

- 让读者 10 秒内知道这段时间主要做了什么
- 不超过 2 条 bullet

### 功能开发生成方法

每个模块建议包含：

1. 模块名
2. 一句话描述
3. 具体内容
4. 涉及页面/功能
5. 图片预留位（如适用）

模块名要用业务语言，例如：

- 搜索能力升级
- 订单详情页优化
- 配置化运营能力接入
- 新增活动页搭建

避免写成：

- 修改 `useXXX.ts`
- 调整 `index.vue`
- 新增 `utils.ts`

### Bug 修复生成方法

只总结重要问题，例如：

- 页面白屏
- 筛选条件失效
- 数据回显错误
- 交互逻辑异常
- 接口容错问题

写法要求：

- 先说明问题
- 再说明解决思路
- 能体现业务影响更好

### 技术优化生成方法

常见归类包括：

- 代码重构
- 性能优化
- 公共组件抽象
- 工程配置优化
- 文档完善

如果优化内容很零散，需要聚合成 1-2 条高层表达，不要写成碎片列表。

## 交互确认流程

第一次生成总结后，用户可以在当前根目录查看到生成内容，且必须给用户一个选择卡片。

使用 `AskQuestion`，提供两个选项：

- `接受保留`
- `继续精简`

如果用户选择 `继续精简`：

1. 提示用户输入想精简的部分
2. 用户可能会描述：
   - “再短一点”
   - “功能点合并一下”
   - “弱化技术描述”
   - “只保留 5 个核心点”
3. 根据用户反馈重新压缩文案
4. 再次展示精简后的 markdown 版本

精简时优先做这些动作：

- 合并相近模块
- 删除低价值补充句
- 弱化技术细节
- 保留模块名、结果、价值、页面范围

## 语雀上传流程

在用户确认最终版本后，再给第二个选择卡片，询问是否上传到语雀。

选项建议：

- `上传到语雀`
- `暂不上传`

### 上传目标

固定上传到以下文档下新建子文档：

`https://hellobike.yuque.com/agta80/hhvky1/wu8xb1n5yid48d65`

新文档标题规则：

- `YYYY年MM月报告`

### 上传前必做

在调用任何 Yuque MCP 工具前，必须先读取对应工具描述文件，确认参数结构。

至少要读取：

- `yuque_get_doc`
- `yuque_toc_query`
- `yuque_create_doc`
- 如涉及覆盖，还要读取 `yuque_update_doc`

### 推荐上传步骤

1. 将最终总结保存为本地 markdown 文件
2. 用 `yuque_get_doc` 读取目标父文档 URL，确认父文档信息
3. 用 `yuque_toc_query` 在知识库 `agta80/hhvky1` 中定位该父文档对应的 TOC 节点
4. 取回父节点 `uuid`
5. 调用 `yuque_create_doc`，使用：
   - `namespace: "agta80/hhvky1"`
   - `title: "YYYY年MM月报告"`
   - `body_file: 本地 markdown 文件路径`
   - `target_uuid: 父节点 uuid`
6. 上传成功后，把返回结果中的文档链接反馈给用户

### 重名处理

如果 `yuque_create_doc` 返回重名提示：

- 不要静默覆盖
- 必须再次使用 `AskQuestion` 让用户选择：
  - `覆盖已有文档`
  - `重新新建`
  - `取消上传`

如果用户选择覆盖：

1. 定位已有同名文档
2. 调用 `yuque_update_doc`
3. 使用相同标题和 `body_file` 更新内容

## 失败处理

### 没有 Git 提交

直接说明：

- 当前时间范围内没有查询到该作者提交
- 无法基于 Git 生成有效总结

### 作者名为空

直接询问用户提供 Git 作者名，不要自己臆测。

### 提交太多且非常分散

优先聚合，不要扩写。

如果仍然难以收敛，先按模块给出 3-4 个主项，再把其他内容折叠为一句“其他零散优化”。

### 语雀上传失败

必须明确说明：

- 失败发生在哪一步
- 失败原因
- 是否已经生成本地 markdown 文件
- 是否需要用户重试或改为手动上传

## 自检清单

输出前检查：

- [ ] 是否先通过 `git config user.name` 获取了作者
- [ ] 是否按用户要求的时间范围获取了提交
- [ ] 是否做了文件类型统计
- [ ] 是否同时参考了 commit message 和 diff
- [ ] 是否过滤了小改动
- [ ] 是否最终只保留 3-4 个主要工作点
- [ ] 是否按“功能开发 / Bug 修复 / 技术优化”分模块整理
- [ ] 是否使用 markdown 输出
- [ ] 是否在初稿后给了“接受保留 / 继续精简”卡片
- [ ] 是否在最终稿后给了“是否上传语雀”卡片
- [ ] 如需上传，是否先读取了 Yuque MCP 工具描述

## 推荐话术

### 第一次总结完成后

“已根据最近 N 天的提交记录整理出精简版总结，主要聚合为 X 个模块。你可以先看下当前版本，如果需要我再继续压缩表达，我可以直接按你的要求精简。”

### 继续精简时

“请告诉我希望精简的方向，比如：更短一些、只保留 5 个点、弱化技术描述、合并相近模块。”

### 上传前

“总结已确认完成。是否需要我继续帮你上传到指定语雀目录？文档标题会按 `YYYY年MM月报告` 创建。”
