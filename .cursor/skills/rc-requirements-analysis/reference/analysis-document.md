# 需求分析文档格式

## 文档命名和存储

### 文档命名

文档名称：`XXX需求分析.md`（`XXX` 为需求简称）

### 存储位置

保存在：`AICodingDoc/xxx需求/` 目录下

## 文档内容要求

需求分析文档应包含两个步骤的分析结果：

### 需求理解与信息收集

1. 仔细阅读产品PRD文档（如果提供）和用户的需求理解
2. 识别需求的核心目标和业务价值
3. 明确各技术栈的分工和协作关系
4. 搜索codebase，定位相关的现有代码和模块

### 改动点识别与分析

1. 系统化分析当前技术栈需要做的所有改动
2. 识别涉及的现有模块、类和方法

## 改动点信息要求

对于每个识别出的改动点，需明确：
- 改动位置（类名、方法名、文件路径）
- 改动类型（新增/修改/删除）
- 改动原因和目的描述，尽量简短
- 改动点的核心逻辑描述

## 改动点聚合原则（重要）

**核心原则**：改动点需要聚合，但要保留技术定位信息，便于后续技术方案设计和编码实现。

### 四个核心要求

1. **按模块/功能聚合**：将技术上相关的多个改动点聚合
   - 同一模块/包的多个类改动 → 聚合为一个改动项
   - 同一功能特性的多层改动（如配置层+服务层+业务层）→ 聚合为一个改动项
   - 例如：配置类 + 配置接口 + 配置实现 → 聚合为"配置管理模块"
   - 例如：Service + DTO + Mapper + Controller → 聚合为"订单服务层改动"

2. **保留技术定位**：改动项应包含技术定位信息
   - ✅ 推荐："配置管理模块（HitchSkuTitlePrefixConfig体系）"
   - ✅ 推荐："订单服务层改动（OrderService、OrderDTO、OrderMapper）"
   - ✅ 推荐："首页Fragment重构（HomeFragment、OrderCardView）"
   - 在改动项名称中标注主要涉及的类/模块

3. **说明改动内容和原因**：既说"做什么"也说"为什么"
   - 需要说清楚：要做什么技术改动（具体的改动内容）
   - 需要说清楚：为什么要这样改（技术原因或业务原因）
   - ✅ 好："新增配置管理模块，实现运营配置的读取、匹配和缓存逻辑，支持按城市、里程等多维度条件灵活控制展示"
   - ❌ 差："新增运营配置能力"（缺少技术信息）
   - ❌ 差："新增HitchSkuTitlePrefixConfig类"（太细碎，未聚合）

4. **控制合理粒度**：改动项一般 5-8 个
   - 不要按每个类拆分（太细碎，不便于Review）
   - 不要聚合成1-2个（太粗糙，无法指导后续实现）
   - 按模块/功能维度聚合

### 聚合思路

当遇到多个技术改动时：
- 识别技术相关性：这些改动是否在同一模块/功能下？
- 按模块功能归类：将技术相关的改动聚合
- 保留技术定位：在改动项中标注主要类/模块/包名
- 说明改动和原因：描述要做什么技术改动，以及为什么这样改

## 维护要求

- 后续如果需求内容变更，也要维护好这个文档
- 这个阶段专注于结合代码现状与文档进行需求分析，理解要做的事情，不需要做任何代码修改变动
