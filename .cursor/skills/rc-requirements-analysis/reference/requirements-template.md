# 需求输入模版

在需求分析阶段开始前，必须按照以下模版要求用户提供需求信息。将此模版完整贴在会话中，让用户可以直接复制并修改。

## 完整模版

```markdown
#### 产品prd链接
这一部分非必须，可以贴链接，如果觉得自己在理解部分可以描述清楚，可以不要
https://hellobike.yuque.com/xxxx
此链接为内网链接，应该通过yuque mcp获取内容

#### 我的理解
xxxx，需要将开发对需求的理解，自己要做的事情进行描述。

示例（服务端）：需要在司机订单发单，修改订单，以及常用路线添加、修改成功之后，调用风控的提供的接口，把添加，修改相关的动作以及关联的数据信息提供给风控，供风控后续执行风控触发动作。

示例（移动端）：首页已经有一个进行中订单存在，在起点输入框的上方，并且功能比较简单，无操作类的功能，本次需要优化，将顶部的进行中订单模块删除，在输入框下放新增一个进行中订单模块，能力更丰富。

#### 各个技术栈大致分工
按如下框架，描述我负责的部分要做哪些事情，与我协作的技术栈比如后端做哪些事情（非必须）
- Android/iOS/前端/小程序
    - xxx
    - xx
    - 示例：负责展示进行中订单卡片
    - 示例：负责处理相关新增点击功能的处理
    - 示例：数据尽可能由服务端提供，尽量不要处理过多逻辑
- 服务端：
    - x
    - 示例：在常用路线，司机临时行程调用选单列表接口时，我需要查询司机订单/司机常用路线上的风控字段（字段由下游提供，你可以先mock一个字段），根据这个风控字段，来决策是否需要调用匹配的订单列表。如果命中风控，则直接返回空列表，如果没有命中则正常选单列表逻辑。
    - 示例：服务端行程中台（其他团队负责）：负责车主发单，修改订单调用风控，并存储风控字段到司机订单上。
    - 示例：负责组装进行中订单数据，返回给端侧

#### 其他补充信息
非必须，有额外补充信息可以在这里描述
```

## 字段说明

### 产品prd链接
- **必填性**：非必须
- **用途**：提供产品需求文档链接
- **说明**：如果是内网语雀链接（hellobike.yuque.com），应通过 yuque mcp 工具获取内容
- **建议**：如果能在"我的理解"部分描述清楚，可以省略此字段

### 我的理解
- **必填性**：必填
- **用途**：开发者对需求的理解和要做的事情的描述
- **要求**：
  - 清晰描述需求目标
  - 说明自己要负责的具体工作，尽可能详细
  - 明确技术实现方向
- **建议**：用具体业务场景描述，避免抽象概括

### 各个技术栈或者各个服务的大致分工
- **必填性**：可选
- **用途**：明确多端协作的职责划分
- **格式**：按技术栈分类列举
  - Android/iOS/前端/小程序
  - 服务端
- **要求**：
  - 明确列出自己负责的部分
  - 可选地说明协作方负责的部分
  - 建议每个职责点用一句话清晰描述

### 其他补充信息
- **必填性**：非必须
- **用途**：补充任何上述字段未覆盖的重要信息
- **内容**：技术约束、时间要求、依赖关系等
