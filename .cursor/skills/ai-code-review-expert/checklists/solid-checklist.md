# SOLID 异味提示词

## SRP（单一职责原则）

- 一个文件/模块承担了不相关的关注点（例如：HTTP + DB + 领域规则写在同一个文件）
- 类/模块体积很大、内聚性低，或者存在多个“变化原因”
- 函数把许多互不相关的步骤强行编排在一起
- “上帝对象”：对系统了解过多、承担过多职责
- **自问**：“这个模块发生变化的唯一原因是什么？”

## OCP（开闭原则）

- 新增一种行为需要修改很多 `switch/if` 分支
- 功能增长依赖“改动核心逻辑”而不是“扩展/插拔”
- 没有策略/插件/Hook 等可变点
- **自问**：“我能否在不改动既有代码的情况下新增一种变体？”

## LSP（里氏替换原则）

- 子类依赖具体类型判断，或对基类方法抛异常/不支持
- 重写方法弱化前置条件或强化后置条件
- 子类忽略父类语义，或把父类行为变成空实现
- **自问**：“调用方在不感知的情况下能否替换为任意子类？”

## ISP（接口隔离原则）

- 接口方法过多，大部分实现类并不需要
- 调用方为了少量能力被迫依赖一个很“胖”的接口
- 出现大量空实现/桩实现（stub）的方法
- **自问**：“所有实现者真的会用到接口的所有方法吗？”

## DIP（依赖倒置原则）

- 高层业务逻辑直接依赖具体的 I/O、存储或网络类型
- 硬编码实现类，而不是依赖抽象/注入
- import 链把业务逻辑与基础设施强耦合在一起
- **自问**：“我能否在不改业务逻辑的情况下替换实现？”

---

## 常见代码异味（不止 SOLID）

| 异味 | 典型征兆 |
|------|----------|
| **过长的方法** | 函数超过 30 行，多层嵌套 |
| **特性依恋** | 方法更多使用“别的类”的数据而非自身数据 |
| **数据簇** | 同一组参数经常成组出现、反复传递 |
| **原始类型偏执** | 用字符串/数字表达领域概念，而非领域类型 |
| **霰弹式修改** | 一处改动需要在很多文件同步改 |
| **发散式变化** | 同一个文件因许多不相关原因频繁变化 |
| **死代码** | 不可达或从未被调用的代码 |
| **过度泛化** | 为“假想未来需求”提前抽象 |
| **魔法数字/字符串** | 硬编码值且缺少命名常量 |

## 重构启发式

1. **按职责拆分，而不是按体积拆分**：小文件也可能违反 SRP
2. **只在需要时引入抽象**：等到第二个用例再抽象
3. **保持重构增量化**：先隔离行为，再移动/重组
4. **先保行为，再动结构**：结构调整前先补测试
5. **按意图命名**：如果命名很难，抽象可能不对
6. **组合优于继承**：继承容易带来紧耦合
7. **让非法状态不可表示**：用类型系统约束不变量
