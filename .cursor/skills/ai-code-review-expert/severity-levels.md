# 问题严重程度定义

> 用于统一团队对代码问题严重程度的判断标准，便于快速对齐处理优先级与合入门槛。

## 📊 严重程度分级

| 级别 | 标识 | 定义 | 合入建议 |
|------|------|------|----------|
| **严重** | 🔴 | 会导致崩溃/安全问题/数据损坏/核心功能不可用，或问题必现 | **必须修复**，否则不允许合入 |
| **中等** | 🟠 | 潜在风险、边界场景问题、明显逻辑缺陷、可预见的稳定性/性能隐患 | **建议修复**；确需先合入要给出影响评估与后续修复计划 |
| **轻微** | 🟡 | 规范性/可维护性/可读性问题，或收益较小的优化建议 | **可选修复**；推荐逐步清理或单独建跟进 |

---

## 🔴 严重问题（必须修复）

### 判定标准

- **稳定性**：必现崩溃、强制类型转换崩溃、数组越界、明显的并发崩溃
- **安全性**：敏感信息泄漏、越权、注入、明文存储敏感数据
- **数据**：会造成数据丢失/错乱（例如写错 key、订单状态不可逆错误）
- **功能**：核心链路不可用或直接阻断（支付、下单、接单、登录等）

## 🟠 中等问题（建议修复）

### 判定标准

- **边界与异常态**：未覆盖失败分支/默认分支，导致特定场景逻辑缺失
- **资源管理**：可预见的内存泄漏、重复注册监听、未释放资源等
- **性能**：热点路径做重计算/主线程阻塞/明显 N+1 查询/无分页加载
- **架构与设计**：明显违反职责划分、依赖方向混乱，后续维护成本高



## 🟡 轻微问题（可选优化）

### 判定标准

- 命名不清晰、可读性较差，但不影响正确性
- 存在魔法数字/字符串、重复代码，可抽取常量或方法
- 结构可以更清晰（函数偏长、嵌套偏深），但当前仍可维护


## ✅ 使用建议

- **优先级**：先找 🔴，再处理 🟠，最后做 🟡
- **记录口径**：严重程度与 `templates/issue-template.md` / `templates/report-template.md` 保持一致
- **不确定时**：按“最坏影响”评估严重程度，并写清触发条件与影响范围
