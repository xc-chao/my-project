# 技术方案文档转换

将需求分析文档和技术方案文档整合转换为符合公司技术方案模版的正式评审文档。

## 快速开始

### 确认输入信息

转换前需要收集：

1. **需求分析文档路径**：通常为 `AICodingDoc/xxx需求/xxx需求分析.md`
2. **技术方案文档路径**：通常为 `AICodingDoc/xxx需求/xxx需求技术方案.md`
3. **需求名称**：用于生成输出文件名
4. **产品PRD链接**（可选）：如需求分析中有提供

### 输出说明

生成文件：`xxx需求-评审版技术方案.md`，保存在与输入文档相同的目录。

## 文档结构模版

生成的文档包含以下章节（严格按公司模版）：

```markdown
# 修订版本
v1.0（默认版本，评审后由用户维护）

# 人员邀请及阅读打卡
（保留模版格式，由用户填写）

# 需求文档
（填写产品PRD链接）

# 需求分析
## 名词解释
（从需求分析文档提取业务名词）

## 用例分析
（选填，如有用例图则呈现）

## 改动点分析
（核心章节，见下方详细规范）

# 变更披露
## 数据变化
待补充

## API变化
（提取接口变更信息或填"无"）

# 技术设计与实现
## 架构设计
（如有架构图则呈现）

## 详细设计
（完整复制技术方案文档内容）

# 技术风险
## 灰度方案
待补充

## 观测方案
待补充

## 应急方案
待补充

# 成本评估
无/待评估

# 技术方案评审记录
（保留模版格式，评审后填写）
```

## 核心转换规范

### 改动点分析表格

这是最重要的章节，必须使用**产品视角**编写。

**表格格式：**

| 改动项 | 改动分类 | 改动类型 | 改动说明 |
|--------|----------|----------|----------|

**字段填写规范：**

**改动项**
- 用简短的名词短语描述功能点
- 例如："运营配置能力"、"订单流程优化"、"首页订单卡片"

**改动分类**
- ✅ 使用业务视角：配置管理、业务流程、业务规则、数据模型、用户交互、接口变更、性能优化、安全加固
- ❌ 避免技术分层：配置层、服务层、UI层、数据层

**改动类型**
- 只能是：新增、修改、删除

**改动说明**
- 公式：现状描述（可选）+ 改动内容 + 业务价值
- 必须包含业务价值/解决的问题/支持的能力
- ✅ 好："新增运营配置能力,支持通过配置中台按城市、里程等多维度灵活控制前缀展示,实现精细化运营"
- ❌ 差："新增HitchSkuTitlePrefixConfig配置类,实现usable()方法进行配置匹配"

### 聚合原则（重要）

改动点必须按功能聚合，控制合理数量：

**按功能聚合**
- 将同一功能相关的多个技术改动（配置类、接口、实现、常量）聚合为一个改动项
- 例如：配置类 + 配置列表类 + 配置接口 + 配置实现 → 聚合为"运营配置能力"

**控制数量**
- 一般 3-5 个，最多不超过 8 个
- 超过 8 个说明聚合不够

**业务视角**
- 问自己：产品会怎么描述这个改动？解决了什么业务问题？

**突出价值**
- 支持了什么能力？
- 解决了什么问题？
- 提升了什么体验？
- 实现了什么目标？

### 自检清单

- [ ] 改动项数量是否合理？（3-5个，最多不超过8个）
- [ ] 是否按功能点聚合，而非技术实现拆分？
- [ ] 改动分类是否使用业务视角？
- [ ] 改动说明是否使用业务语言，避免技术术语？
- [ ] 改动说明是否包含业务价值/解决的问题/支持的能力？
- [ ] 产品经理/测试人员能否快速理解每个改动项？

## 详细设计章节处理

**直接照搬输入技术方案文档的详细设计部分，保持原有内容、格式、图表完全不变。**

包括但不限于：
- 现状逻辑描述
- 改动方案说明
- 实现流程/步骤
- Mermaid 流程图
- 代码示例
- 接口定义
- 数据结构

无需做任何转换或优化，完整复制即可。

## 工作流程

### 第一步：读取输入文档

读取需求分析文档，提取：
- 改动点信息
- 需求理解和业务名词
- 产品PRD链接（如有）

读取技术方案文档，提取：
- 详细设计部分
- 架构图、流程图等图表
- 接口定义、数据结构等技术细节

### 第二步：转换需求分析部分

**名词解释**
- 从需求分析文档识别业务名词
- 如没有明确定义，标注"无"

**用例分析**
- 如技术方案中有用例图或用例描述，在此呈现
- 否则标注"本需求无复杂业务场景，不需要用例分析"

**改动点分析（核心）**
- 将需求分析文档中的改动点转换为表格形式
- 根据技术栈选择合适的改动分类
- **用产品视角的业务语言改写改动说明**

### 第三步：转换技术设计与实现部分

**架构设计**
- 如技术方案中有架构图，完整复制
- 如有架构说明文字，一并复制
- 如没有架构相关内容，标注"本需求为功能优化，无架构变更"

**详细设计**
- 直接照搬输入技术方案文档的详细设计部分
- 保持原有内容、格式、图表完全不变
- 无需做任何转换或优化

### 第四步：填充其他章节

- **需求文档**：填写产品PRD链接
- **变更披露**：
  - 数据变化：标注"待补充"
  - API变化：如技术方案中有接口变更，提取相关信息；否则填"无"
- **技术风险**：灰度、观测、应急三个部分都标注"待补充"
- **成本评估**：标注"无"或"待评估"
- **其他模版章节**：保留原始格式，由用户后续填写

### 第五步：生成完整文档

1. 按照公司技术方案模版的完整结构组织内容
2. 确保所有章节标题、格式与模版一致
3. 保留必要的模版提示文字（如"必填"、"选填"等）
4. 输出文档名为 `xxx需求-评审版技术方案.md`
5. 保存到与输入文档相同的目录下

## 注意事项

1. **详细设计直接照搬**：技术设计与实现部分完整复制输入技术方案文档内容，不做任何转换
2. **改动点表格转换 - 产品视角**：
   - 改动说明必须使用产品视角的业务语言
   - 描述业务价值和用户体验，而非技术实现细节
   - 让产品经理能直接理解改动目的
3. **格式规范**：严格遵守公司技术方案模版的格式要求
4. **待补充标注**：对于需要用户后续填写的部分，明确标注"待补充"
