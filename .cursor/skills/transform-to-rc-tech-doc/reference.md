# 技术方案文档转换参考指南

本文档提供更详细的转换参考信息，包括改动分类参考、业务语言词汇表等。

## 改动分类参考

选择合适的改动分类时，优先使用业务视角的分类，避免技术架构分层。

### 推荐的改动分类

| 分类 | 适用场景 | 示例 |
|------|----------|------|
| **配置管理** | 新增或修改配置能力、配置中台集成 | 运营配置能力、灰度配置、AB实验配置 |
| **业务流程** | 改变业务处理流程、流程优化 | 询价流程优化、订单流程改造、支付流程调整 |
| **业务规则** | 修改业务逻辑、判断条件、规则引擎 | 定价规则调整、优惠规则优化、匹配规则改进 |
| **数据模型** | 新增或修改数据结构、数据库表 | 订单数据模型扩展、用户画像数据增强 |
| **用户交互** | UI/UX改进、页面调整、交互优化 | 首页卡片优化、表单交互改进、弹窗提示优化 |
| **接口变更** | API接口新增、修改、废弃 | 接口升级v2、新增查询接口、接口参数调整 |
| **性能优化** | 性能提升、资源优化、加载优化 | 列表加载优化、缓存策略优化、内存优化 |
| **安全加固** | 安全相关改进、权限控制 | 数据加密、权限校验、防刷机制 |
| **监控告警** | 监控埋点、日志、告警 | 业务埋点增强、异常监控、性能监控 |
| **容错处理** | 异常处理、降级、兜底 | 接口降级策略、异常兜底逻辑、超时处理 |

### 避免使用的技术分层分类

| ❌ 技术分层分类 | 原因 |
|----------------|------|
| 配置层 | 过于技术化，产品无法理解 |
| 服务层 | 只描述技术架构，不体现业务价值 |
| UI层 | 应改为"用户交互"等业务视角描述 |
| 数据层 | 应改为"数据模型"或"接口变更" |
| 业务逻辑层 | 应改为具体的业务分类如"业务流程"、"业务规则" |
| 基础设施层 | 过于底层，产品不关心 |

## 业务语言词汇表

编写改动说明时，使用业务语言而非技术术语。

### 配置相关

| ❌ 技术术语 | ✅ 业务语言 |
|------------|------------|
| 新增Config类 | 新增配置能力 |
| 实现配置读取接口 | 支持配置动态读取 |
| 配置key常量定义 | 支持配置中台管理 |

### 流程相关

| ❌ 技术术语 | ✅ 业务语言 |
|------------|------------|
| 修改processOrder方法 | 优化订单处理流程 |
| 增加参数传递 | 增强流程数据传递 |
| 调整方法调用顺序 | 调整业务处理顺序 |

### UI相关

| ❌ 技术术语 | ✅ 业务语言 |
|------------|------------|
| 修改Fragment布局 | 优化页面布局 |
| 新增onClick监听 | 支持用户点击操作 |
| 调整View位置 | 调整UI展示位置 |

### 接口相关

| ❌ 技术术语 | ✅ 业务语言 |
|------------|------------|
| 调用新API | 升级数据接口 |
| 修改请求参数 | 增强数据查询能力 |
| 增加响应字段 | 返回更丰富的数据 |

### 数据相关

| ❌ 技术术语 | ✅ 业务语言 |
|------------|------------|
| 新增数据库字段 | 扩展数据模型 |
| 修改数据结构 | 优化数据结构 |
| 增加缓存逻辑 | 提升数据访问性能 |

## 改动说明编写模板

### 模板结构

```
[改动内容] + [业务价值/解决的问题/支持的能力]
```

### 模板示例

**配置管理类**
```
新增[功能名称]配置能力,支持通过配置中台[配置维度说明],实现[业务目标]
```
- 示例：新增运营配置能力,支持通过配置中台按城市、里程等多维度灵活控制前缀展示,实现精细化运营

**业务流程类**
```
优化[流程名称],通过[优化手段],提升[性能指标/用户体验]
```
- 示例：优化询价流程,通过按需读取配置策略（开关关闭时不读取配置列表）,提升流程性能

**业务规则类**
```
调整[规则名称],从[旧规则]改为[新规则],支持[业务场景]
```
- 示例：调整标题前缀展示规则,从无条件展示改为按配置动态控制,支持精细化运营和AB实验

**用户交互类**
```
优化[页面/组件],增加[新功能/新展示],提升[用户体验维度]
```
- 示例：优化首页订单卡片,增加订单详情展示和快捷操作入口,提升用户操作便捷性

**接口变更类**
```
升级[接口名称]从[旧版本]到[新版本],获取[更丰富的数据],支持[新功能需求]
```
- 示例：升级订单接口从v1到v2,获取起终点、费用、司机信息等详细数据,支持首页卡片完整展示

**性能优化类**
```
优化[功能模块],通过[优化手段],提升[性能指标]
```
- 示例：优化列表加载,通过分页加载和图片懒加载,提升首屏加载速度50%

## 聚合策略参考

### 聚合维度

按以下维度识别可聚合的改动点：

1. **功能维度**：实现同一功能的所有改动聚合
   - 配置类 + 接口 + 实现 + 常量 → "运营配置能力"

2. **用户场景维度**：服务同一用户场景的改动聚合
   - UI改动 + 数据改动 + 逻辑改动 → "首页订单卡片优化"

3. **业务流程维度**：同一流程的多个环节改动聚合
   - 询价入口 + 询价流程 + 询价结果展示 → "询价流程优化"

### 聚合判断标准

**可以聚合的情况**：
- [ ] 服务同一业务目标
- [ ] 产品视角看是同一个功能
- [ ] 测试时会作为一个功能点验证
- [ ] 上线时会同时发布

**不应聚合的情况**：
- [ ] 属于不同业务模块
- [ ] 解决不同的产品需求
- [ ] 可以独立上线
- [ ] 面向不同的用户场景

### 聚合案例分析

**案例1：配置能力**

技术实现涉及：
- HitchSkuTitlePrefixConfig.java（配置类）
- HitchSkuTitlePrefixConfigList.java（配置列表类）
- OperatePlatformService.java（接口定义）
- OperationPlatformServiceImpl.java（接口实现）
- Consts.java（配置key常量）

聚合结果：
- **改动项**：运营配置能力
- **改动分类**：配置管理
- **改动类型**：新增
- **改动说明**：新增运营配置能力,支持通过配置中台控制顺风车标题前缀展示。包含功能总开关和按城市、里程、同跨城、预约/实时、估价来源等多维度的灵活配置规则,支持"任一匹配即生效"逻辑和AB实验

**案例2：首页改造**

技术实现涉及：
- HomeFragment.java（UI逻辑）
- home_layout.xml（布局文件）
- OrderRepository.java（数据层）
- OrderApi.java（接口定义）
- OrderCardAdapter.java（卡片适配器）

聚合结果：
- **改动项**：首页订单卡片优化
- **改动分类**：用户交互
- **改动类型**：修改
- **改动说明**：优化首页订单卡片展示,将卡片位置调整至输入框下方,增加订单详情、状态、操作按钮等功能,并升级数据接口获取完整订单信息,提升用户操作便捷性和信息可见性

- 业务流程（服务流程）
- 业务规则（业务逻辑）
- 接口变更（API提供）
- 数据模型（数据库表）
- 性能优化（查询优化、缓存）

常用分类：
- 用户交互（页面UI）

- 接口变更（API调用）
- 性能优化（加载优化、渲染优化）
- 配置管理（运营配置）

## 常见问题与解决方案

### Q1：改动点太多怎么办？

**问题**：技术实现涉及20+个文件，改动点数量难以控制。

**解决方案**：
1. 先识别产品需求涉及几个功能点（通常3-5个）
2. 按功能点将技术改动归类
3. 每个功能点作为一个改动项
4. 技术细节放在"详细设计"章节

### Q2：如何判断是否过于技术化？

**判断标准**：
- 产品经理能否理解？
- 测试人员能否据此设计测试用例？
- 改动说明是否包含业务价值？

**自查方法**：
- 把改动说明读给非技术人员听
- 如果需要解释类名、方法名才能理解，说明过于技术化

### Q3：新增功能和修改功能如何区分？

**新增**：
- 之前完全没有这个能力
- 全新的功能模块
- 示例：新增运营配置能力、新增订单快捷操作

**修改**：
- 原有功能的优化、增强、调整
- 改变现有的业务规则或流程
- 示例：优化首页卡片展示、调整定价规则

### Q4：配置类改动如何描述？

**不好的描述**：
- 新增HitchSkuTitlePrefixConfig配置类

**好的描述**：
- 新增运营配置能力,支持通过配置中台动态控制XX展示规则

**关键**：
- 强调"能力"而非"类"
- 说明配置支持的业务场景
- 体现配置带来的灵活性和价值

### Q5：多个模块都有改动如何聚合？

**示例场景**：首页、询价页、订单详情页都需要调整UI

**聚合策略**：
1. 如果是为了同一个产品需求，聚合为一个改动项
   - 改动项：XX功能多页面适配
2. 如果是不同的产品需求，拆分为多个改动项
   - 改动项1：首页XX优化
   - 改动项2：询价页XX优化

## 检查清单

转换完成后，使用此清单检查质量：

### 改动点分析检查

- [ ] 改动项数量合理（3-5个，最多不超过8个）
- [ ] 改动项名称使用业务语言（无类名、方法名、技术术语）
- [ ] 改动分类使用业务视角（无技术分层术语）
- [ ] 改动类型准确（仅限"新增"、"修改"、"删除"）
- [ ] 改动说明包含业务价值
- [ ] 改动说明长度适中（1-3句话）
- [ ] 产品经理能够理解每个改动项
- [ ] 测试人员能够据此设计测试用例

### 文档完整性检查

- [ ] 修订版本已填写（默认v1.0）
- [ ] 需求文档链接已填写（或标注"无"）
- [ ] 名词解释已完成（或标注"无"）
- [ ] 用例分析已完成（或标注"不需要"）
- [ ] API变化已说明（或标注"无"）
- [ ] 详细设计已完整复制
- [ ] 所有流程图、代码示例已保留
- [ ] 待补充项已标注"待补充"

### 格式规范检查

- [ ] 所有章节标题符合模版要求
- [ ] 表格格式正确
- [ ] Markdown语法正确
- [ ] 代码块格式正确
- [ ] 列表格式规范

## 转换速查表

| 转换环节 | 关键原则 | 常见错误 | 正确做法 |
|---------|---------|---------|---------|
| 改动项命名 | 业务语言 | 使用类名、方法名 | 使用功能点名称 |
| 改动分类 | 业务视角 | 配置层、UI层 | 配置管理、用户交互 |
| 改动说明 | 业务价值 | 只描述技术动作 | 说明解决的问题和带来的价值 |
| 改动聚合 | 功能维度 | 按文件拆分 | 按功能聚合 |
| 详细设计 | 完整复制 | 重新编写、简化 | 保持原有内容不变 |
