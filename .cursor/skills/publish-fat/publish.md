# Publish FAT Workflow

## 适用场景

当用户表达以下意图时使用本技能：

- 发布fat环境
- 发布到fat环境
- 发布fat
- fat发布
- 发布 FAT

目标是基于当前仓库分支，按规范完成 `fat` 环境发布，并在发布前处理本地未提交代码、提交确认、推送确认和发布确认。

## 执行原则

- 默认使用项目级 Skill：`.cursor/skills/publish-fat/`
- 发布前必须先检查当前分支和 `git status`
- 需要用户确认时，优先使用 `AskQuestion`，以可点击选项卡形式收集答案
- 未经用户明确确认，不得执行 `git commit`
- 未经用户明确确认，不得执行 `git push`
- 发布 MCP 调用前，必须先读取工具描述文件，确认参数结构
- 发布成功前，不得向用户声称“已发布”
- 如果遇到 MCP 服务异常、协议异常或权限异常，必须如实说明阻塞点

## 确认交互规范

### 必须使用卡片选择

当需要用户确认时，优先调用 `AskQuestion`，不要让用户手动输入：

- `是`
- `否`
- `继续`
- `取消`

除非 `AskQuestion` 不可用，才退回普通文本询问。

### 卡片设计规则

- 每次只问一个关键问题
- 每个问题默认单选
- 选项文案要直接表达动作，不要只写“是/否”
- 选项数量控制在 2 到 3 个
- 如果存在风险分支，选项文字里直接写明后果

推荐选项风格：

- `提交当前改动并继续`
- `不提交，停止发布`
- `push 当前分支并继续发布`
- `不 push，但继续发布远端代码`
- `取消本次发布`

## 标准流程

### 1. 初始检查

先做以下检查：

1. 获取当前分支：`git branch --show-current`
2. 检查工作区状态：`git status --short`
3. 如需发布，读取 deploy MCP 工具描述：
   - `build_web_app.json`

向用户同步的信息应包含：

- 当前分支名
- 是否存在未提交代码
- 接下来要进入“提交确认”还是“发布确认”

### 2. 有未提交代码时

如果 `git status` 显示有未提交内容，按以下流程执行：

1. 明确告知用户存在未提交代码，不能直接发布
2. 使用 `AskQuestion` 询问用户是否提交当前改动
3. 如果用户不同意提交，停止发布流程
4. 如果用户同意提交：
   - 先查看 `git status`、`git diff`、`git log`
   - 拟定简洁准确的提交信息
   - 执行提交
5. 提交完成后，使用 `AskQuestion` 询问用户是否将当前分支推送到远端

推荐卡片：

问题：

- `检测到当前分支存在未提交代码。请选择下一步操作。`

选项：

- `提交当前改动并继续`
- `不提交，停止发布`

提交完成后的推荐卡片：

问题：

- `当前改动已提交。请选择是否 push 当前分支后再发布 fat。`

选项：

- `push 当前分支并继续发布`
- `不 push，继续下一步确认`
- `取消本次发布`

### 3. 用户拒绝 push 时

如果用户拒绝 `push`，必须明确提醒：

- 发布平台通常基于远端分支构建
- 未 push 的本地提交不会进入本次 fat 发布

然后继续使用 `AskQuestion` 确认一次：

问题：

- `如果不 push，本次发布通常只会使用远端当前分支代码，不包含你刚刚的本地提交。是否仍然继续发布 fat？`

选项：

- `继续发布远端当前代码`
- `取消本次发布`

处理规则：

- 用户确认继续：继续发 fat，但要明确说明发布的可能是远端旧代码
- 用户不确认继续：停止发布

### 4. 没有未提交代码时

如果工作区干净，不要直接发布，先使用 `AskQuestion` 做一次明确确认：

问题：

- `当前分支工作区干净。是否直接发布当前分支到 fat 环境？`

选项：

- `发布当前分支到 fat`
- `取消本次发布`

处理规则：

- 用户确认：继续发布
- 用户拒绝：停止发布

## 发布执行

### 1. 参数准备

调用 MCP 前，准备以下参数：

- `projectName`: 优先使用项目工程名，例如 `AppRcpVue3Web`
- `branch`: 当前分支，例如 `release/all`
- `env`: 固定为 `fat`
- `lastTag`: 传空字符串 `""`

### 2. MCP 调用要求

必须先读取工具描述，再调用：

- MCP 服务：`user-deploy-mcp-server`
- Tool：`build_web_app`

调用参数示例：

```json
{
  "projectName": "AppRcpVue3Web",
  "branch": "release/all",
  "env": "fat",
  "lastTag": ""
}
```

### 3. 调用后的反馈

如果发布成功，返回给用户的信息至少应包含：

- 发布环境：`fat`
- 发布分支
- 平台返回的关键信息
- 如果结果中包含部署链接，直接告诉用户可以点击查看

如果发布失败，必须说明：

- 是参数问题、平台业务错误，还是 MCP 服务异常
- 如果能定位具体原因，要直接给出下一步建议

## 异常处理

### 1. MCP 工具调用异常

如果出现类似以下问题：

- `Invalid tools/call result`
- MCP 返回内容格式异常
- 工具调用成功但客户端无法解析

优先排查：

1. `mcp.json` 中 `deploy-mcp-server` 是否正确配置了 stdio 启动参数
2. 服务是否把日志输出到了标准输出，导致 MCP 协议被污染
3. 当前 MCP 服务是否可正常启动并暴露 `build_web_app`

### 2. 远端分支风险

如果本地有提交但未 push，需要明确告诉用户：

- 发布平台大概率读取远端分支
- 本地未推送提交不会进入发布结果

### 3. 不要绕过确认

以下动作都需要明确确认：

- 提交代码
- 推送远端
- 真正执行 fat 发布

确认方式优先级：

1. 使用 `AskQuestion`
2. 如果 `AskQuestion` 不可用，再退回普通文本确认

## 输出风格

对用户的输出应保持简洁、明确、可执行：

- 先说结论，再说下一步
- 只问一个关键确认问题，避免连续堆叠多个问题
- 在提交、推送、发布三个关键节点分别确认

## 推荐执行模板

### 模板 A：发现未提交代码

优先调用 `AskQuestion`：

- title: `发布前确认`
- question id: `confirm_commit`
- prompt: `检测到当前分支有未提交代码，不能直接发布 fat。请选择下一步操作。`
- options:
  - `commit_and_continue`: `提交当前改动并继续`
  - `stop_publish`: `不提交，停止发布`

文本降级方案：

```text
检测到当前分支有未提交代码，不能直接发布 fat。
请确认是否先提交当前改动；确认后我再继续检查是否需要 push 并发起发布。
```

### 模板 B：提交完成后确认 push

优先调用 `AskQuestion`：

- title: `推送确认`
- question id: `confirm_push`
- prompt: `当前改动已提交。请选择是否 push 当前分支后再发布 fat。`
- options:
  - `push_and_publish`: `push 当前分支并继续发布`
  - `skip_push_confirm_publish`: `不 push，继续下一步确认`
  - `cancel_publish`: `取消本次发布`

```text
当前改动已提交。
是否继续 push 当前分支到远端，再发布 fat？
```

### 模板 C：工作区干净时确认发布

优先调用 `AskQuestion`：

- title: `发布确认`
- question id: `confirm_publish_clean`
- prompt: `当前分支工作区干净。是否直接发布当前分支到 fat 环境？`
- options:
  - `publish_now`: `发布当前分支到 fat`
  - `cancel_publish`: `取消本次发布`

```text
当前分支工作区干净。
是否直接发布当前分支到 fat 环境？
```

### 模板 D：拒绝 push 后再次确认

优先调用 `AskQuestion`：

- title: `风险确认`
- question id: `confirm_publish_without_push`
- prompt: `如果不 push，本次 fat 发布通常不会包含你本地最新提交，只会使用远端当前分支代码。是否仍然继续发布 fat？`
- options:
  - `publish_remote_code`: `继续发布远端当前代码`
  - `cancel_publish`: `取消本次发布`

```text
如果不 push，本次 fat 发布通常不会包含你本地最新提交，只会使用远端当前分支代码。
是否仍然继续发布 fat？
```

## 完成标准

满足以下条件才算完成本技能流程：

- 已检查当前分支
- 已检查 `git status`
- 涉及提交时拿到了用户确认
- 涉及 push 时拿到了用户确认
- 涉及发布时拿到了用户确认
- 已读取 `build_web_app` 工具描述
- 已调用或明确停止调用 `user-deploy-mcp-server`
- 已将成功结果或失败原因明确反馈给用户
