# 禁止事项清单

在所有阶段和文档中，**严格禁止**以下内容。

## ❌ 时间相关估算

### 禁止内容
- 工作量评估（如"预计3天"、"大约8小时"）
- 时间估算（如"本周完成"、"下周上线"）
- 任务时长标注（如"2小时完成XX功能"）

### 正确做法
- 专注于任务内容本身，不提及时间
- 按优先级排序任务，不按时间排序
- 使用任务完成状态标记（`- [ ]` / `- [x]`）

### 示例对比

**❌ 错误**：
```markdown
## 任务拆解
- [ ] 实现用户登录功能（预计2天）
- [ ] 编写单元测试（预计4小时）
```

**✅ 正确**：
```markdown
## 任务拆解
- [ ] 实现用户登录功能
- [ ] 编写单元测试
```

## ❌ 技术风险章节

### 禁止内容
- 单独的"技术风险"章节
- "风险评估"部分
- "潜在问题"罗列

### 原因
技术方案应专注于"怎么做"，而不是"可能出什么问题"。风险识别是项目管理范畴，不属于技术方案文档。

### 可接受的替代
- 在改动点描述中，简要说明"为什么这样设计"
- 在技术选型中，说明"选择A而非B的原因"

## ❌ 观测方案章节

### 禁止内容
- "监控指标"章节
- "日志方案"章节
- "告警配置"章节

### 原因
观测是运维范畴，技术方案应专注于功能实现本身。

### 例外情况
- 如果需求明确要求"增加XX监控指标"，则将其作为一个功能点处理
- 云端相关需求可以简要说明关键配置项

## ❌ 应急预案章节

### 禁止内容
- "回滚方案"章节
- "降级策略"章节
- "应急响应"章节

### 原因
应急预案是发布管理范畴，不属于技术方案设计。

## ❌ 上线计划章节

### 禁止内容
- "发布步骤"章节
- "上线时间表"
- "发布检查清单"

### 原因
上线计划是项目管理范畴，技术方案应专注于技术实现。

## ❌ 测试相关内容

### 禁止内容
- "单元测试设计"章节
- "测试用例"章节
- "测试验证"章节
- "自测准出标准"
- 详细的测试场景罗列

### 原因
测试相关内容不属于技术方案设计阶段，技术方案应专注于详细设计本身。

## ❌ 依赖关系章节

### 禁止内容
- "新增依赖"章节
- "服务依赖"章节
- "第三方库版本"章节
- "依赖关系图"

### 原因
依赖关系管理是实施阶段的产物，技术方案应专注于设计本身。如果需要引入新依赖，在实施阶段直接添加即可。

## ❌ 后续优化建议

### 禁止内容
- "后续优化"章节
- "优化方向"章节
- "TODO"章节
- "未来规划"章节

### 原因
技术方案应该是完整的、可落地的方案，不应该包含"待定"或"后续优化"的内容。如果有优化想法，要么现在就做，要么不在方案中提及。

## 总结：专注于什么

技术方案应该专注于：

✅ **做什么**：需求分析，明确改动点  
✅ **怎么做**：技术方案，详细设计  
✅ **分步骤**：任务拆解，可执行清单  
✅ **写代码**：代码实现，逐步完成  

❌ **不要关注**：时间、风险、监控、应急、上线、测试用例、依赖关系、后续优化

**记住**：保持技术方案的纯粹性，专注于技术设计本身。
