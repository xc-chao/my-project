# 系统总体架构与技术选型

## 1 系统总体设计思想

本系统的设计目标是实现一套面向移动端用户的智能购物小程序，使用户能够在微信小程序环境中完成商品浏览、搜索、咨询、加购、下单、订单查看和售后申请等操作。为了保证系统结构清晰、后续扩展方便，系统采用前后端分离的总体架构。前端负责页面展示、用户交互、表单输入、状态维护和接口调用；后端负责业务规则处理、权限校验、数据持久化、AI 服务代理和统一异常处理；数据库负责保存系统运行过程中产生的用户、商品、购物车、订单、售后、反馈和聊天记录等数据。

从项目代码结构来看，前端工程位于 `frontend/miniprogram` 目录下，核心页面集中在 `src/pages`，公共组件集中在 `src/components`，接口请求封装集中在 `src/services`，状态管理集中在 `src/store`。后端工程位于 `backend` 目录下，核心服务位于 `backend/src`，入口文件 `app.js` 中挂载了认证、商品、购物车、订单、聊天、地址、售后、反馈和后台管理等多个业务路由。系统整体结构能够较好地体现“页面展示层、接口服务层、业务逻辑层、数据访问层”的分层思想。

## 2 前端技术选型

前端部分采用 `uni-app` 作为小程序开发框架。`uni-app` 能够使用 Vue 风格语法进行页面开发，并支持编译到微信小程序等多端环境，适合毕业设计中快速构建移动端业务页面。本系统中，页面路由统一配置在 `frontend/miniprogram/src/pages.json` 中，包括登录页、首页、搜索页、商品详情页、聊天页、确认订单页、订单列表页、地址页、售后页、反馈页、后台管理页、购物车页和个人中心页等。

前端页面采用自定义导航样式，整体风格保持一致。系统设置了四个底部 Tab：首页、搜索、购物车、我的，对应用户购物过程中的主要入口。页面组件方面，项目提供了 `AppHeader`、`ProductCard`、`PillTabBar`、`AddressCard`、`ChatBubble`、`OrderCard`、`QuantityStepper`、`EmptyStateCard` 等组件，用于抽象常见页面结构和交互单元。通过组件复用，可以减少页面间重复代码，也能使页面风格保持一致。

状态管理方面，系统采用 Pinia 管理用户信息和购物车信息。`user` 模块用于维护当前用户登录状态、用户角色、头像、昵称等信息；`cart` 模块用于维护购物车商品列表、商品数量和选中状态。接口调用方面，前端将业务请求集中封装在 `services` 目录，例如 `authService.ts`、`productService.ts`、`cartService.ts`、`orderService.ts`、`chatService.ts`、`adminService.ts` 等。这样页面层不需要直接拼接接口地址，而是通过语义化的方法调用后端服务，提高了代码可读性和可维护性。

## 3 后端技术选型

后端采用 `Node.js + Express` 实现接口服务。Node.js 具有事件驱动、非阻塞 I/O 的特点，适合处理小程序场景中大量接口请求。Express 框架轻量、灵活，能够快速搭建 RESTful API 服务，适合本系统这种模块较多但业务复杂度适中的项目。

在 `backend/src/app.js` 中，系统创建 Express 应用，并使用 `helmet`、`cors`、`express.json`、`express-session` 等中间件。其中，`helmet` 用于增强基础 HTTP 安全性，`cors` 用于跨域访问控制，`express.json` 用于解析 JSON 请求体，`express-session` 用于维护服务端会话。系统还挂载了请求 ID 中间件、可选认证中间件、统一 404 处理和统一错误处理中间件，使接口具备较完整的基础能力。

后端业务模块包括：

- `auth`：用户登录、退出、资料获取和资料更新。
- `product`：商品列表、商品详情、分类和搜索。
- `cart`：购物车查询、添加、修改和删除。
- `order`：订单预览、订单创建和订单列表。
- `chat`：AI 会话创建、消息发送和历史记录。
- `address`：收货地址增删改查。
- `afterSale`：售后申请和售后记录。
- `feedback`：商品反馈和用户反馈。
- `admin`：后台概览、商品、订单、售后、反馈管理。

后端路由统一挂载在 `/api` 前缀下，例如认证接口挂载到 `/api/auth`，购物车接口挂载到 `/api/cart`，订单接口挂载到 `/api/orders`，聊天接口挂载到 `/api/chat`。这种接口组织方式符合 RESTful 风格，也便于前端按业务域进行调用。

## 4 数据库与持久化设计

系统采用 MySQL 作为核心数据库。购物系统具有明显的结构化数据特征，例如用户资料、商品信息、购物车条目、订单主表、订单明细、地址信息、售后申请、反馈记录和聊天记录等，这些数据之间存在较明确的关联关系，适合使用关系型数据库进行管理。

数据库连接与事务封装位于 `backend/src/config/db.js`。该文件提供了连接池获取、普通查询、事务处理、数据库存在性检查和连接关闭等函数。对于订单创建、库存扣减、购物车清理等需要保证原子性的业务，系统可以通过事务封装统一处理，从而避免部分操作成功、部分操作失败导致的数据不一致问题。

数据库表设计遵循业务域拆分原则，建议至少包括：

- `users`：保存用户身份、昵称、头像、角色等信息。
- `products`：保存商品标题、价格、库存、分类、标签等信息。
- `product_galleries`：保存商品图集。
- `cart_items`：保存购物车条目。
- `addresses`：保存用户收货地址。
- `orders` 与 `order_items`：保存订单主信息和订单商品明细。
- `order_timeline` 与 `order_shipments`：保存订单状态流转和物流信息。
- `after_sales`：保存售后申请。
- `feedbacks`：保存用户反馈。
- `chat_sessions` 与 `chat_messages`：保存 AI 会话和聊天消息。

## 5 系统分层结构

本系统后端模块普遍采用 `routes -> controller -> service -> repository` 的结构。`routes` 层负责定义接口路径和绑定中间件；`controller` 层负责接收请求参数并调用服务；`service` 层负责处理业务规则；`repository` 层负责数据库读写。这种结构能够避免控制器中堆积复杂业务逻辑，也便于后续单独测试业务服务或替换数据存储方式。

前端则采用 `page -> component -> service -> request` 的依赖方向。页面层主要负责页面状态、生命周期和用户交互；组件层负责可复用 UI 单元；服务层负责具体业务接口；请求层负责统一请求、错误处理和响应解析。通过这样的分层方式，页面代码不会直接处理底层网络细节，系统整体维护成本较低。

## 6 本章小结

本章从系统目标、前端技术、后端技术、数据库设计和分层架构等方面介绍了系统总体设计。系统采用 uni-app、Node.js、Express 和 MySQL 构建主体能力，并在后端集成 AI 服务。该架构既能满足毕业设计中完整功能展示的需求，也具有较好的扩展空间。
