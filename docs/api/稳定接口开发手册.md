# 稳定接口开发手册

## 文档目标

这份文档用于解决当前项目的三个核心问题：

1. 前端页面已经完成，但后端真实接口尚未系统开发
2. 商品详情页和搜索页存在“字段不够、交互无效”的问题
3. 前后端还缺少一套可持续扩展的接口开发方法

本手册以 `Express.js + 模块化分层 + 契约优先` 为原则，目标不是“先把接口凑出来”，而是让接口可以长期稳定演进。

## 一、接口开发总原则

### 1. 契约优先

任何接口编码前，先确定 4 件事：

1. 页面真实需要什么字段
2. 字段的数据类型、是否必填、默认值是什么
3. 字段是展示字段、业务字段还是控制字段
4. 前端点击某个交互后，后端应该如何响应

结论：

- 前端不要靠文案硬编码业务状态
- 后端不要返回“页面自己猜”的数据结构
- 接口先写文档和校验，再写控制器和服务层

### 2. 模块优先

接口必须按业务模块开发，不按“页面”零散开发。

建议模块：

- `auth`
- `product`
- `search`
- `cart`
- `order`
- `address`
- `afterSale`
- `chat`

禁止做法：

- 在一个大路由文件里堆所有接口
- 控制器里直接写数据库逻辑
- 前端页面对多个无约束接口做临时拼装

### 3. 向后兼容优先

在真实接口替换 mock 阶段，优先遵守：

- 新增字段可以，删除字段要评估影响
- 字段类型不要反复变动
- 列表接口统一分页
- 筛选和排序统一走 query 参数
- 错误结构统一

## 二、统一接口规范

## 1. 路由规范

当前项目已经使用 `/api` 前缀，建议正式接口继续沿用，并在上线前统一收敛。

推荐路径：

- `GET /api/products`
- `GET /api/products/:id`
- `GET /api/cart`
- `POST /api/cart/items`
- `GET /api/orders`
- `POST /api/orders`
- `GET /api/addresses`
- `POST /api/chat/sessions`

说明：

- 搜索不要长期独立成“临时接口”，建议收敛到 `GET /api/products`
- 所有搜索、筛选、排序都走 query 参数
- 一个资源只保留一组主要入口，避免 `/search` 和 `/products` 两套并存太久

### 2. 成功返回结构

建议所有接口统一为：

```json
{
  "success": true,
  "message": "ok",
  "data": {},
  "requestId": "req_xxx"
}
```

列表接口建议：

```json
{
  "success": true,
  "message": "ok",
  "data": {
    "list": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 100,
      "hasMore": true
    }
  },
  "requestId": "req_xxx"
}
```

### 3. 失败返回结构

```json
{
  "success": false,
  "code": "VALIDATION_ERROR",
  "message": "参数不合法",
  "errors": [],
  "requestId": "req_xxx"
}
```

### 4. 错误码建议

- `VALIDATION_ERROR`
- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `CONFLICT`
- `RATE_LIMIT_EXCEEDED`
- `INVENTORY_NOT_ENOUGH`
- `INTERNAL_SERVER_ERROR`

## 三、接口开发流程

每个模块统一按以下顺序开发：

1. 补页面文档，明确字段和交互
2. 写接口文档，明确 request/response
3. 写 `validator`
4. 写 `repository`
5. 写 `service`
6. 写 `controller`
7. 写 `routes`
8. 写模块集成测试
9. 前端切换真实接口
10. 删除对应 mock 分支

### 开发完成定义

一个接口开发完成，至少满足：

- 路由可访问
- 参数可校验
- 错误有统一返回
- 单元测试或集成测试通过
- 前端对应交互不再依赖硬编码

## 四、当前必须补齐的接口字段

这里直接对应你现在前端的两个问题。

## 1. 商品详情页字段不够

当前 `product/detail.vue` 只有：

- 标题
- 副标题
- 价格
- 原价
- 库存
- 销量
- 尺码
- 图集

但页面已经出现或即将出现的内容包括：

- `48h 上新`
- `95% 好评`
- 服务承诺
- 热卖/新品等营销标签
- 推荐信息

### 建议详情接口

`GET /api/products/:id`

### 建议返回结构

```json
{
  "id": "p_001",
  "title": "Nike Air Jordan 1 Low",
  "subtitle": "黑白低帮",
  "price": 1899,
  "originalPrice": 2199,
  "stock": 36,
  "sales": 2300,
  "category": "鞋靴",
  "cover": "/static/xxx.jpg",
  "gallery": ["/static/1.jpg", "/static/2.jpg"],
  "badges": ["热门上新"],
  "sizes": ["39", "40", "41", "42", "43"],
  "detail": "经典低帮轮廓，适合通勤与日常穿搭。",
  "marketing": {
    "isNewIn48h": true,
    "praiseRate": 95,
    "hotScore": 92
  },
  "serviceTags": [
    "48h上新",
    "95%好评",
    "7天无理由",
    "AI尺码建议"
  ],
  "shipping": {
    "shipWithinHours": 24,
    "supportAfterSale": true
  }
}
```

### 前端使用规则

- `48h 上新` 不应该写死在首页或详情页里
- `95% 好评` 不应该是纯静态文案
- 应由 `marketing` 和 `serviceTags` 驱动展示

### 详情页要新增的交互字段

如果后面还要扩展，建议预留：

- `favoriteCount`
- `commentCount`
- `skuOptions`
- `recommendProducts`

## 2. 搜索页筛选无效

当前搜索页的：

- `综合`
- `价格`
- `热门`
- 筛选项

本质上只是静态 UI，没有驱动真实查询。

### 建议搜索接口统一收敛

`GET /api/products`

### 推荐 query 参数

```text
keyword=
page=1
pageSize=20
sort=comprehensive|priceAsc|priceDesc|hot
category=鞋靴,服饰
isNewIn48h=true
minPrice=0
maxPrice=2000
```

### 推荐返回结构

```json
{
  "list": [],
  "pagination": {
    "page": 1,
    "pageSize": 20,
    "total": 50,
    "hasMore": true
  },
  "searchMeta": {
    "keyword": "Jordan 1",
    "sort": "comprehensive",
    "availableSorts": [
      { "key": "comprehensive", "label": "综合" },
      { "key": "priceAsc", "label": "价格从低到高" },
      { "key": "priceDesc", "label": "价格从高到低" },
      { "key": "hot", "label": "热门" }
    ],
    "availableFilters": {
      "categories": ["鞋靴", "服饰", "配件"],
      "flags": [
        { "key": "isNewIn48h", "label": "48h上新" }
      ]
    }
  }
}
```

### 前端交互映射

- 点击 `综合`：`sort = comprehensive`
- 点击 `价格`：在 `priceAsc / priceDesc` 间切换
- 点击 `热门`：`sort = hot`
- 点击分类或筛选项：写入 query，重新请求列表
- 搜索结果页不应该只改前端高亮状态，必须重新触发接口

## 六、后端文件模板

每个业务模块保持同样结构：

```text
modules/product/
├── product.routes.js
├── product.controller.js
├── product.service.js
├── product.repository.js
└── product.validator.js
```

### 职责边界

- `routes`：只负责路由声明、中间件拼装
- `controller`：只负责收参、调用 service、返回 response
- `service`：只负责业务逻辑
- `repository`：只负责数据读写
- `validator`：只负责参数规则

禁止把业务判断写进：

- `routes`
- `controller`

## 七、测试与验收

### 1. 每个模块至少验证

- 正常请求
- 参数错误
- 未登录访问
- 资源不存在
- 空列表
- 边界值

### 2. 商品详情模块重点测试

- 有 `marketing` 字段时前端显示 48h 上新和好评率
- 无 `marketing` 字段时页面降级不报错
- `gallery` 为空时主图仍可正常显示

### 3. 搜索模块重点测试

- `综合 / 价格 / 热门` 均能触发真实排序
- 分类与价格区间筛选能组合
- 分页切换不丢失筛选状态

## 八、近期开发顺序建议

建议按这个顺序推进，最稳定：

1. 先重构 `product` 模块，统一列表与详情字段
2. 再完成 `search` 的 query 化与排序筛选
3. 再完成 `cart + order`
4. 然后补齐 `address + afterSale`
5. 最后完成 `chat`

这样能先解决你当前最明显的问题：

- 商品详情字段不足
- 搜索交互无效
- mock 和真实接口结构不一致
