# 模块化开发实施手册

## 文档目标

这份文档不再讨论“接口地址怎么写”，而是回答另一个更重要的问题：

如何把当前这个全栈项目拆成稳定、可协作、可扩展的模块，避免后面越写越乱。

当前项目已经具备：

- `uni-app` 前端页面雏形
- `Express.js` 后端基础框架
- 页面文档与设计稿

下一步应该把“页面工程”升级成“模块工程”。

## 一、模块化开发的核心原则

### 1. 先按业务域拆，再按技术层拆

不要先按“页面文件多不多”拆模块，而是先按业务域拆：

- 用户认证
- 商品与搜索
- 购物车
- 订单
- 地址
- 售后
- AI 问答
- 个人中心

然后在每个业务域内部，再拆技术层：

- 前端页面层
- 前端组件层
- 前端状态层
- 前端服务层
- 后端路由层
- 后端控制器层
- 后端业务层
- 后端数据层

### 2. 依赖方向必须单向

推荐依赖关系：

```text
页面 page
  -> 业务组件 business components
  -> store / service
  -> request
  -> backend api

route
  -> controller
  -> service
  -> repository
```

禁止反向依赖：

- 页面直接操作底层请求细节
- 组件内部直接调用接口
- controller 直接访问数据库
- service 依赖页面字段命名

### 3. 一个模块只关心一个业务闭环

例如：

- `product` 模块只负责商品列表、商品详情、商品属性、推荐商品
- `search` 模块只负责关键词、排序、筛选、结果列表
- `order` 模块只负责下单、预览、列表、状态流转

不要把“猜你喜欢、聊天入口、推荐逻辑、营销 banner”全部混进同一个页面文件里处理。

## 二、项目推荐模块结构

## 1. 前端模块结构

建议在现有 `frontend/miniprogram/src` 内继续稳定为以下层次：

```text
src/
├── pages/                # 页面容器，只处理页面级状态和路由
├── components/
│   ├── common/           # 通用组件
│   ├── business/         # 业务组件
│   └── layout/           # 布局组件
├── services/             # 接口调用封装
├── store/                # Pinia 状态
├── mock/                 # mock 数据与 mock handler
├── constants/            # 常量与枚举
├── utils/                # 通用工具
├── types/                # 领域模型类型
└── static/               # 本地静态资源
```

### 页面层职责

- 接收路由参数
- 组织页面布局
- 触发加载
- 维护页面级交互状态

页面层不应该负责：

- 直接拼接口 URL
- 直接定义业务枚举
- 直接写复杂数据转换逻辑

### 组件层职责

建议进一步拆出：

- `components/common`：按钮、空状态、步进器、卡片基础件
- `components/business/product`：商品图集、营销标签、SKU 选择器
- `components/business/search`：排序栏、筛选栏、搜索结果摘要
- `components/business/order`：订单卡、商品行、地址摘要

## 2. 后端模块结构

当前后端已经接近模块化，可继续强化为：

```text
backend/src/
├── common/
├── config/
├── middleware/
├── integrations/
└── modules/
    ├── auth/
    ├── product/
    ├── search/
    ├── cart/
    ├── order/
    ├── address/
    ├── afterSale/
    └── chat/
```

### 每个模块固定五件套

- `*.routes.js`
- `*.controller.js`
- `*.service.js`
- `*.repository.js`
- `*.validator.js`

这样做的好处：

- 新人接手时位置固定
- 测试边界清晰
- 代码评审标准统一

## 三、优先级最高的业务模块拆分

这里直接对应你现在最急的页面问题。

## 1. 商品模块 `product`

### 模块职责

- 首页商品推荐
- 商品详情
- 商品营销属性
- 商品图集
- 商品推荐关系

### 当前问题

- 详情页结构偏轻，只能显示基础字段
- 缺少 `48h 上新`、`95% 好评` 等营销属性
- 页面里的展示信息没有沉淀成结构化字段

### 前端建议拆分

`pages/product/detail.vue` 不要长期维持成一个大文件，建议拆为：

- `ProductGallery`
- `ProductSummary`
- `ProductMarketingTags`
- `ProductServiceTags`
- `ProductSkuSelector`
- `ProductPurchaseBar`

### 后端建议拆分

后端 `product` 模块至少输出四类数据：

- 基础信息
- 营销信息
- 服务承诺
- 推荐信息

### 状态模型建议

前端详情页建议维护：

- `product`
- `selectedSku`
- `quantity`
- `previewImage`
- `loading`
- `recommendList`

## 2. 搜索模块 `search`

### 模块职责

- 搜索关键词输入
- 排序切换
- 筛选条件
- 搜索结果统计
- 搜索结果列表

### 当前问题

- `综合`、`价格`、`热门` 等只是静态切换
- 页面有视觉样式，但没有业务状态机
- 结果页和首页推荐之间没有统一排序/筛选规则

### 前端建议拆分

`pages/search/index.vue` 建议拆成：

- `SearchBar`
- `SearchSortBar`
- `SearchFilterPanel`
- `SearchSummary`
- `SearchResultGrid`
- `SearchEmptyState`

### 建议状态模型

```ts
type SearchState = {
  keyword: string;
  page: number;
  pageSize: number;
  sort: 'comprehensive' | 'priceAsc' | 'priceDesc' | 'hot';
  category: string[];
  isNewIn48h: boolean;
  minPrice?: number;
  maxPrice?: number;
};
```

### 交互规则

- 点击 `综合`：重置为默认排序
- 点击 `价格`：升序和降序切换
- 点击 `热门`：按热度或销量排序
- 修改任一筛选条件：重置 `page = 1`
- 关键词变化时：清空旧结果并重新拉取

## 3. 购物车模块 `cart`

### 模块职责

- 购物车商品列表
- 勾选状态
- 数量变更
- 规格变更
- 去结算

### 前端建议拆分

- `CartBanner`
- `CartList`
- `CartItemCard`
- `CartSummaryBar`
- `CartEmptyState`

### 后端要点

- 数量更新要做库存校验
- 下架商品要能返回失效状态
- 勾选状态可前端持有，也可后端持久化，二选一，不要混用

## 4. 订单模块 `order`

### 模块职责

- 订单预览
- 创建订单
- 订单列表
- 订单详情
- 状态流转

### 状态流转建议

至少定义：

- `pendingPayment`
- `paid`
- `shipping`
- `completed`
- `cancelled`
- `afterSale`

### 前端建议拆分

- `OrderPreviewCard`
- `OrderGoodsList`
- `OrderPriceBreakdown`
- `OrderStatusTabs`
- `OrderCard`

## 5. 地址模块 `address`

### 模块职责

- 地址列表
- 新增地址
- 编辑地址
- 删除地址
- 默认地址切换

### 注意点

- 默认地址只能有一个
- 下单页优先读取默认地址
- 删除默认地址时要给出补偿策略

## 6. 售后模块 `afterSale`

### 模块职责

- 发起售后申请
- 售后记录列表
- 售后状态展示

### 推荐状态

- `reviewing`
- `approved`
- `rejected`
- `completed`

## 7. AI 模块 `chat`

### 模块职责

- 基于商品上下文发起对话
- 保存会话
- 限流
- 超时与降级

### 模块边界

AI 模块只回答：

- 尺码建议
- 材质说明
- 发货说明
- 售后说明

不要让 AI 模块直接修改订单、地址、库存等真实业务数据。

## 8. 管理后台模块 `admin`

### 模块职责

- 管理员概览看板
- 商品管理入口
- 订单处理入口
- 用户反馈与评价查看
- Agent 调用监控

### 权限要求

- 只有 `admin` 身份能看到入口和访问接口
- 前端做可见性控制
- 后端做 `requireAuth + requireAdmin` 双重校验

### 前端建议拆分

- `AdminOverview`
- `AdminMetricGrid`
- `AdminModuleList`
- `AdminFeedbackPanel`
- `AdminMonitorPanel`

### 后端建议拆分

- `GET /api/admin/overview`
- `GET /api/admin/products`
- `GET /api/admin/orders`
- `GET /api/admin/feedback`

## 四、共享模块设计

## 1. 类型模块

建议新增 `src/types`，统一维护：

- `Product`
- `ProductDetail`
- `SearchQuery`
- `SearchMeta`
- `CartItem`
- `Order`
- `Address`
- `AfterSaleRecord`

这样可以解决当前一个常见问题：

前端 mock 类型、页面字段、后端响应字段名字不统一。

## 2. 常量模块

建议新增：

- `constants/sort.ts`
- `constants/order.ts`
- `constants/product.ts`

例如搜索页的排序，不要散落成多个中文字符串硬编码。

## 3. 状态模块

Pinia 里建议只放“跨页面共享状态”：

- 用户登录态
- 购物车数量
- 搜索历史
- 当前会话上下文

页面私有状态仍然留在页面内部。

## 五、推荐开发顺序

为了避免“页面一大堆，但接口和状态都没沉淀”的情况，建议按下面顺序推进。

## 第一阶段：统一类型和契约

- 把商品、搜索、订单、地址类型统一
- 补完页面文档
- 补完接口文档

## 第二阶段：先打通商品与搜索

原因：

- 首页、搜索页、详情页共用同一批商品数据
- 这是当前最明显的缺口
- 能最快把 mock 数据升级成真实业务数据

本阶段完成后应达到：

- 首页商品来自真实接口
- 搜索排序和筛选可用
- 商品详情可展示营销属性和服务属性

## 第三阶段：打通交易链路

- 购物车
- 订单预览
- 创建订单
- 地址选择

## 第四阶段：补齐售后和 AI

- 售后申请
- AI 咨询
- 会话历史

## 六、模块开发完成标准

每个模块上线前，至少满足以下要求：

### 1. 文档完整

- 有页面文档
- 有接口文档
- 有字段说明

### 2. 代码边界清晰

- 页面不直接请求底层
- 组件不直接耦合页面逻辑
- service 不关心 UI 展示
- controller 不写业务细节

### 3. 可测试

- 至少有基础集成测试
- 异常路径可验证
- 空状态可验证

### 4. 可替换 mock

- mock 数据结构与真实接口结构一致
- 切换真实接口时页面不需要大改

## 七、针对你当前项目的直接落地建议

如果只看现在最需要做的事，建议立刻按下面执行：

1. 先把 `product` 和 `search` 合并成一个核心商品域来治理
2. 给详情页补 `marketing`、`serviceTags`、`shipping` 等字段
3. 给搜索页补统一的 `sort + filters + pagination` 状态模型
4. 把搜索页当前无效点击改造成“状态变化 -> 重新请求列表”
5. 再启动 `cart + order + address` 的真实接口开发

这样做的结果是：

- 页面不会继续靠假数据“硬撑”
- 后端接口不会和页面样式脱节
- 后续每个模块都能独立开发和验收
