# 技能调度规则

本项目的 `.kiro/skills/` 目录下有 14 个技能文件，它们已作为 workspace 规则自动加载到上下文中。你必须根据以下调度规则主动遵循对应技能的完整流程。

## 强制调度矩阵

| 用户意图                 | 必须遵循的技能                                 | 执行顺序                                 |
| ------------------------ | ---------------------------------------------- | ---------------------------------------- |
| 开发新功能/新页面/新组件 | brainstorming → writing-plans → 实现           | 先问清需求，再出设计，用户批准后才写代码 |
| 修复 bug / 测试失败      | systematic-debugging → test-driven-development | 先找根因，再写失败测试，再修复           |
| 实现已有计划             | executing-plans 或 subagent-driven-development | 按计划逐步执行                           |
| 代码审查相关             | requesting-code-review / receiving-code-review | 完成功能后请求审查，收到反馈后技术验证   |
| 声称工作完成             | verification-before-completion                 | 必须先运行验证命令，有证据才能声称完成   |

## 核心行为规则

1. **brainstorming 是入口**：任何涉及"开发"、"创建"、"添加"、"修改功能"的请求，必须先走 brainstorming 流程——一次问一个问题，理解需求后提出 2-3 个方案，用户批准设计后才进入实现。
2. **不要跳过设计直接写代码**：即使需求看起来很简单，也必须先确认设计。
3. **TDD 贯穿始终**：写任何生产代码前必须先有失败的测试。
4. **完成前必须验证**：不能说"应该可以了"，必须运行命令并展示输出。
5. **一次只问一个问题**：brainstorming 阶段每条消息只问一个问题，优先用选择题。
